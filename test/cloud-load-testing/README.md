# 云压测服务对比

## 1. 阿里云 PTS（Performance Testing Service）

### 优势
- ✅ 分布式压测，支持百万级并发
- ✅ 真实公网 IP，模拟全国各地用户
- ✅ 支持 HTTP/HTTPS/WebSocket/gRPC
- ✅ 自动生成压测报告
- ✅ 与阿里云监控集成

### 使用步骤
1. 登录阿里云 PTS 控制台
2. 创建压测场景
3. 配置压测参数：
   - 目标 URL: http://your-gateway:8888/seckill/v1/seckill
   - 并发用户数: 10000
   - 压测时长: 10 分钟
   - 地域分布: 华北、华东、华南
4. 启动压测
5. 查看实时监控和报告

### 定价
- 按 VUM（Virtual User Minutes）计费
- 1000 VUM ≈ ¥10
- 10000 并发 × 10 分钟 = 100000 VUM ≈ ¥1000

### 适用场景
- 生产环境验证
- 大促前压测
- 性能基准测试

## 2. 腾讯云 WeTest

### 优势
- ✅ 支持移动端 + Web 压测
- ✅ 真实设备测试
- ✅ 全球节点分布

### 定价
- 按小时计费
- 1000 并发 × 1 小时 ≈ ¥500

---

## 3. AWS CloudWatch Synthetics

### 优势
- ✅ 全球 20+ 区域
- ✅ 与 AWS 生态集成
- ✅ 支持自定义脚本（Node.js/Python）

### 定价
- 按 Canary 运行次数计费
- 1000 次运行 ≈ $0.0012/次

---

## 4. Loader.io（免费方案）

### 优势
- ✅ 免费版支持 10000 并发
- ✅ 简单易用
- ✅ 无需安装

### 限制
- ❌ 免费版只能测试 1 分钟
- ❌ 需要验证域名所有权

### 使用步骤
1. 注册 https://loader.io
2. 验证域名（在网站根目录放置验证文件）
3. 创建测试
4. 查看报告

---

## 5. BlazeMeter（JMeter 云服务）

### 优势
- ✅ 基于 JMeter，兼容现有脚本
- ✅ 支持 100 万+ 并发
- ✅ 全球 56 个数据中心

### 定价
- 免费版: 50 并发
- 专业版: $99/月，1000 并发

## 真实场景模拟清单

### ✅ 必须模拟
1. **真实网络延迟**（10~100ms）
2. **分布式客户端**（不同 IP、不同地域）
3. **完整请求链路**（gateway → seckill-service → order-service）
4. **JWT 认证**（真实 token）
5. **并发连接数**（10000+ HTTP 连接）

### ✅ 建议模拟
6. **不同设备**（PC、手机、平板）
7. **不同浏览器**（Chrome、Safari、Firefox）
8. **网络波动**（丢包、抖动）
9. **CDN 缓存**（静态资源）
10. **数据库主从延迟**（读写分离）

### ⚠️ 可选模拟
11. **DDoS 攻击**（恶意流量）
12. **爬虫流量**（非人类行为）
13. **热点数据**（某个商品特别火）

## 压测检查清单

### 压测前
- [ ] 备份数据库
- [ ] 关闭非必要服务
- [ ] 增大系统资源限制（ulimit、max_connections）
- [ ] 准备监控工具（Grafana、Prometheus）
- [ ] 通知相关人员（避免误报警）

### 压测中
- [ ] 实时监控 CPU、内存、网络、磁盘
- [ ] 观察日志（错误日志、慢查询）
- [ ] 记录关键指标（TPS、延迟、错误率）
- [ ] 截图保存（作为证据）

### 压测后
- [ ] 清理测试数据
- [ ] 分析瓶颈（CPU？内存？IO？网络？）
- [ ] 生成压测报告
- [ ] 制定优化方案
- [ ] 回归测试（验证优化效果）

## 压测报告模板

```markdown
# 秒杀系统压测报告

## 测试环境
- 服务器配置: 8C16G
- 数据库: MySQL 8.0
- 缓存: Redis 6.2
- 消息队列: RabbitMQ 3.9

## 测试场景
- 并发用户数: 10000
- 压测时长: 10 分钟
- 目标 TPS: 5000

## 测试结果
- 实际 TPS: 4500
- 平均延迟: 120ms
- P95 延迟: 350ms
- P99 延迟: 500ms
- 错误率: 0.1%

## 瓶颈分析
1. MySQL 连接池耗尽（300 → 500）
2. Redis 单线程瓶颈（考虑集群）
3. RabbitMQ 消息堆积（增加 consumer）

## 优化建议
1. 增大 MySQL 连接池
2. Redis 集群分片
3. 增加 order-service 实例

## 结论
系统在 4500 TPS 下稳定运行，满足预期目标。
```
